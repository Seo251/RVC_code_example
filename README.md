## 🏗️ RVC 센서 모듈 설계 방식 비교 (Architecture Comparison)

로봇 청소기(RVC) 센서 데이터를 처리하는 두 가지 아키텍처 패턴의 장단점 비교입니다.

### 1. Example 1: 클래스 분리형 (Decoupled Design)
> **센서 통신(하드웨어)과 데이터 변환(소프트웨어)을 분리한 객체지향 설계**

| 장점 (Pros) | 단점 (Cons) |
| :--- | :--- |
| **유지보수 우수:** 센서 부품(초음파 → 라이다 등)이 바뀌어도 변환 로직 코드는 단 한 줄도 수정할 필요가 없음 | **초기 복잡도 증가:** 관리해야 할 클래스(인터페이스)와 파일 개수가 늘어남 |
| **높은 테스트 용이성:** 실제 하드웨어 보드 없이 '가짜 센서 객체'를 주입해 PC에서 소프트웨어 로직만 완벽하게 테스트 가능 | **메모리 오버헤드:** 객체 생성 및 포인터 참조로 인해 아주 미세한 성능/메모리 비용 발생 |
| **재사용성:** 독립된 `ObstacleSensor` 클래스를 고장 진단 등 다른 모듈에서 그대로 가져다 쓸 수 있음 | **학습 곡선:** 의존성 주입(DI)이나 집합(Aggregation) 등 객체지향 개념에 대한 이해가 필요함 |

### 2. Example 2: 클래스 통합형 (Integrated Design)
> **하나의 클래스가 하드웨어 제어부터 데이터 변환까지 모두 책임지는 실용적 설계**

| 장점 (Pros) | 단점 (Cons) |
| :--- | :--- |
| **단순하고 직관적:** 핀 번호 할당부터 결과 데이터 포맷팅까지 코드의 전체 흐름을 한곳에서 바로 파악 가능 | **낮은 확장성:** 하드웨어 사양이나 통신 방식이 조금만 바뀌어도 클래스 내부 통신 로직을 전부 뜯어고쳐야 함 |
| **성능/메모리 최적화:** 객체 간 포인터 참조나 불필요한 인스턴스 생성 과정이 없어 저사양 초소형 MCU 환경에 유리 | **테스트의 한계:** 물리적인 하드웨어 센서가 연결되어 있지 않으면 코드의 정상 작동 여부를 검증하기가 매우 까다로움 |
| **빠른 개발 속도:** 복잡한 구조 설계나 계층 나누기 고민 없이, 빠르고 즉각적인 기능 구현이 가능 | **클래스 비대화:** 센서 종류나 기능이 추가될수록 코드가 길어지고 유지보수가 힘들어짐 (단일 책임 원칙 위배) |
